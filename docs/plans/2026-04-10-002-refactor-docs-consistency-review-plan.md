**Deepened on:** 2026-04-10
**Research agents used:** Architecture Strategist, Pattern Recognition Specialist, Code Simplicity Reviewer, CONTRIBUTING.md Best Practices, Heading Anchor Migration, Vale Prose Linting, Docusaurus v2 SEO/Frontmatter

### Key improvements from deepening

1. **Eliminated three YAGNI items** that would have added significant churn with no functional benefit: the `docs/style-guide.md` duplicate file, `description` frontmatter across 35+ pages, and the `typescript` ‚Üí `ts` global rename (77 occurrences, cosmetic only).

## Overview

A comprehensive review of all documentation in the React Starter Docusaurus site to enforce consistent language, terminology, and navigation. The site has grown organically and shows significant inconsistencies between older and newer content: British vs American English, first-person vs second-person voice, title case vs sentence case headings, mixed code block language identifiers, terminology drift (e.g., "Zengenti Base Package" vs "CRB"), and `sidebar_position` collisions that produce unpredictable ordering. This plan addresses all of these systematically across five phases, from establishing standards through applying them file by file.

## Problem Statement

The docs site covers the `@zengenti/contensis-react-base` (CRB) React Starter ‚Äî a project bootstrapping developers onto the Contensis CMS. Inconsistency in the documentation erodes trust and creates confusion for external developers. Specific issues identified:

### Language and Voice

- 67 occurrences of first-person "we/our" (older docs). Newer docs consistently use second-person "you".
- Mixed UK/US English: "Optimising", "behaviour", "colours", "authorised", "Dockerised" coexist with "standardized", "utilizing", "customized" in newer content.
- "compliment" used where "complement" is correct (`introduction.md`).
- Headings inconsistently apply Title Case vs Sentence case (no documented standard).

### Terminology

- "Elastic Search" (two words) in `introduction.md` ‚Äî should be "Elasticsearch".
- "Zengenti Base Package" in `composer.md` ‚Äî should be "Contensis React Base (CRB)".
- `onRouteLoad` vs `OnRouteLoad` ‚Äî PascalCase headings in `with-events.md` conflict with camelCase usage everywhere else.
- "Approches" typo in `basics/mappers.md`.
- "preferrably" and "seperating" typos in `advanced/composer.md`.
- `canvas.md` contains a second "it's own" (possessive error) ‚Äî same as the known `blocks.md` instance.

### Navigation and Frontmatter

- `sidebar_position` collisions: all deployment docs at position 1; all tips files at position 1. Docusaurus resolves ties alphabetically ‚Äî order is unintentional.
- Three files have **no frontmatter at all**: `routing/with-events.md`, `advanced/tips/forms.md`, `advanced/tips/optimising-large-entries.md`.
- No `title` frontmatter on most doc pages ‚Äî Docusaurus falls back to the `# H1` for the `<title>` tag and `og:title`, with no explicit override available.
- Two pages both titled "Overview" ‚Äî `routing/routing.md` (H1) and `search/search.md` (frontmatter `title`) ‚Äî creates an SEO collision.
- `ci-cd.md` is an empty stub ‚Äî contains only a heading.

### Code Blocks

- `jsx` language identifier used on 18 TypeScript/TSX code blocks (`jsx` does not highlight TypeScript syntax). **Exception**: `canvas.md` line 53 contains a legitimately pure-JSX block ‚Äî do not convert this one.
- Three shell code fence identifiers in use: `sh` (4, all in `forms.md`), `bash` (7), `shell` (2, in `blocks.md`).
- Missing `title=` attribute on code blocks in older docs.

### Content Errors

- Broken link in `deployment/blocks.md` ~line 135: `[*https://www.website.com/...](url),*`.
- Possessive "it's" in `deployment/blocks.md` AND `advanced/canvas.md`: "use it's own routing/documentation".
- `:::caution` "under construction" banner in `routing/routing.md`.
- `:::warning` admonition in `advanced/delivery-api.md` ‚Äî not an official v2.4 admonition type (use `:::caution`).
- Raw HTML `<aside>` wrappers in `advanced/authentication.md` (lines 135 and 235) ‚Äî copy-paste artefact from Notion, not Docusaurus syntax.

### Scope note

`docs/node-starter/getting-started.md` is explicitly out of scope for this review. If its inconsistencies (Title Case headings, "Gitlab" capitalisation, missing `title` frontmatter) are to be addressed, a separate plan should be created.

## Proposed Solution

Five sequential phases:

1. **Style Guide** ‚Äî establish and document the standards in `CONTRIBUTING.md`.
2. **Frontmatter and Navigation** ‚Äî fix position collisions, add missing frontmatter.
3. **Language and Terminology** ‚Äî apply voice, English locale, and glossary corrections across all files. Heading case changes are the final sub-task within this phase.
4. **Code Block Standardisation** ‚Äî normalise language identifiers and `title=` attributes.
5. **Content Completion** ‚Äî address stubs, errors, and broken content.

## Technical Approach

### Architecture

The site is **Docusaurus v2.4.0** using `@docusaurus/preset-classic`. Sidebar is a hybrid: top-level sections are explicit in `sidebars.js`; leaf directories use `type: 'autogenerated'` relying on `sidebar_position` frontmatter. Navigation order in autogenerated sections is determined solely by `sidebar_position`; collisions are resolved alphabetically.

The `onBrokenLinks` setting is **`"throw"`** (not `"warn"`) ‚Äî broken internal links **halt the build**. Any cross-link conversion that introduces a malformed path will fail the build immediately.

No search plugin is configured. No versioning. No `title` or `description` frontmatter is currently in use on most pages.

### Implementation Phases

#### Phase 1: Establish Style Guide (CONTRIBUTING.md)

Create a `CONTRIBUTING.md` at the repo root containing both contributor workflow guidance and the writing style guide. No separate `docs/style-guide.md` is needed ‚Äî that would be a hidden, non-discoverable duplicate. The `CONTRIBUTING.md` is where contributors look.

**Recommended CONTRIBUTING.md sections** (based on Expo, dbt Labs, and Next.js patterns):

1. **Welcome** ‚Äî scope of contributions; one-sentence commits welcome.
2. **Prerequisites and local setup** ‚Äî `git clone` ‚Üí `npm install` ‚Üí `npm start` ‚Üí `http://localhost:3000`.
3. **Repository structure** ‚Äî annotated file tree of `docs/`, what not to touch.
4. **Content types** ‚Äî when to write a tutorial vs guide vs reference vs conceptual page.
5. **Writing style** ‚Äî the standards table below, plus a link to the Google Developer Documentation Style Guide as the catchall for anything not explicitly covered.
6. **Markdown and MDX conventions** ‚Äî required frontmatter, admonition usage, code block rules.
7. **Submitting a pull request** ‚Äî branch naming, CI requirements, review process.

**Standards table to embed in CONTRIBUTING.md:**

| Dimension | Standard | Concrete examples |
|---|---|---|
| Language | American English | "color" not "colour", "standardize" not "standardise", "license" not "licence" |
| Voice | Second person ("you") throughout | "You can configure‚Ä¶" not "We recommend configuring‚Ä¶" |
| Tense | Present tense | "This hook returns‚Ä¶" not "This hook will return‚Ä¶" |
| Heading case | Sentence case | "Getting started" not "Getting Started"; "TypeScript support" not "TypeScript Support" |
| Code symbol headings | Preserve canonical casing | `## onRouteLoad` (code symbol) not `## Onrouteload`; this is distinct from sentence case |
| Avoid qualifiers | Never use: easy, simple, just, quickly, straightforward | "Open the file" not "Simply open the file" |
| Code identifiers | `tsx` for TypeScript+JSX; `ts` for TypeScript-only; `bash` for shell | All new blocks; existing `typescript` blocks may remain |
| Admonitions | `:::info` for terminology; `:::tip` for best practices; `:::caution` for gotchas; `:::danger` for destructive actions | No `:::warning` (v2.4 not officially supported) |
| Cross-links | Relative paths with `.md` extension | `[Routing](../routing/routing.md)` not `/docs/react-starter/routing/routing` |
| Frontmatter | `sidebar_position` and `title` required on every doc page | `title` controls `<title>` tag and `og:title` separately from H1 |

> **On American English:** This is a consistency standard, not a cultural preference. If you write "colour" or "behaviour", reviewers will catch it and suggest the correction ‚Äî no awkwardness intended.

**Terminology glossary to include in CONTRIBUTING.md:**

| Term | Correct usage | Common errors to avoid |
|---|---|---|
| **React Starter** | The starter project (this repository) | "starter project", "the boilerplate" |
| **Contensis React Base (CRB)** | The underlying framework package `@zengenti/contensis-react-base` | "Zengenti Base Package", "the base" without expansion |
| **component** | A React `.component.tsx` file in `src/app/components/` | Do not conflate with "CMS Component" |
| **CMS Component** | An embedded field schema in Contensis (no `sys.id`) | Not the same as a React component |
| **mapper** | A pure function transforming a CMS `Entry` to typed props | Not "transformer" |
| **Composer** | The Contensis Composer field type | Not "page builder" or "composer" (lowercase) |
| **template** | A page-level component in `src/app/templates/` | Distinct from a React component |
| **`onRouteLoaded`** | The correct event hook name (past tense) | `onRouteLoad` (without `-d`) causes a crash |
| **Elasticsearch** | The search engine (one word) | "Elastic Search", "ElasticSearch" |

#### Phase 3: Language and Terminology

**Phase 3 internal ordering ‚Äî critical:**

Complete and commit in this order within Phase 3:
1. Voice changes (first-person ‚Üí second-person) ‚Äî largest prose edits
2. Spelling corrections (British ‚Üí American)
3. Terminology corrections (Elasticsearch, CRB, possessives, typos)
4. **Heading case changes last** ‚Äî run `npm run write-heading-ids` first (see Pre-flight below), then change headings. This ensures anchor IDs are frozen as explicit `{#id}` markers before any heading text changes.

**Pre-flight for heading changes:**

Before editing any heading text, run:
```bash
npm run write-heading-ids
```
This stamps every heading with an explicit `{#anchor-id}` using its current slug, making all subsequent heading text changes anchor-safe. Commit this as a separate "chore: freeze heading anchor IDs" commit before heading edits begin.

> **Note on anchor safety**: Docusaurus uses `github-slugger` which lowercases heading text before generating anchors. Pure Title Case ‚Üí Sentence case changes (only lowercasing mid-word capitals) are safe ‚Äî the lowercase forms are identical. However, `npm run write-heading-ids` is the belt-and-suspenders approach and removes the need to reason about this case-by-case.

## Future: Prose Automation (Post-Review Follow-On)

After this review establishes a clean baseline, add two CI checks to prevent regression. This is a separate piece of work but is documented here to capture the concrete configuration.

Install `markdownlint-cli2` and configure `MD044` to enforce correct product name capitalisation:

```json
// .markdownlint.json
{
  "default": true,
  "MD013": false,
```

Add to `package.json`: `"lint:md": "markdownlint-cli2 \"docs/**/*.md\""`. This runs in under a second and catches terminology capitalisation errors automatically.

## System-Wide Impact

### Interaction Graph

`sidebars.js` drives the sidebar ‚Äî changes to `sidebar_position` frontmatter only take effect when the affected directory uses `type: 'autogenerated'`. Phase 2 changes to `advanced/` top-level `sidebar_position` values will be silently ignored under the current config. Only `deployment/` and `advanced/tips/` have real autogenerated collisions.

### Error and Failure Propagation

`onBrokenLinks` is set to `"throw"` ‚Äî broken internal links **halt the build**, not just produce warnings. Converting cross-links from absolute to relative paths during Phase 5 will immediately surface any currently-broken paths at the next `docusaurus build`. This is desirable as a latent-defect detector. Run the build after each file converted, not in a batch.

### State Lifecycle Risks

The `plans/` exclusion added to `docusaurus.config.js` will cause any plan page URL to 404. Plans were never in the sidebar, so external links are unlikely ‚Äî but confirm before applying.

`draft: true` on `ci-cd.md` makes the page inaccessible in production (404). This is the correct behaviour for an empty stub.

### Integration Test Scenarios

- After Phase 2 position fixes: verify sidebar order in dev server matches the intended sequence for `deployment/` and `advanced/tips/` sections.
- After Phase 3 heading case changes (preceded by `write-heading-ids`): run `docusaurus build` to confirm no broken fragment link errors.
- After Phase 5 link conversions: each converted file must pass an individual `docusaurus build` check before committing the next file.
